在学校服务器上源码编译安装Mysql-8.0.25并运行

一、创建需要的目录

连接NJUPT即可在自己的电脑上访问上课给的服务器，我懒得写一大堆命令，在MacOS系统的Transmit App里连接了服务器，之间图形界面把题目要求的文件夹建好了。

二、cmake编译

老师说服务器上的cmake版本不够编译mysql-8.0.25，有cmake3.22的安装包但不是默认环境。

总之先打开终端

1、cmake编译

我先copy一下PPT里给的命令看看什么效果。学校的服务器自带cmake，没有的可以自己配置一下。

没有出现报错，那接着往下，执行make -j 64。

这个报错出现了超多次，使用cmake -version看了一下版本是3.22。g++ -version看g++版本是10.0.1。

他妈的什么申必服务器，Google解决方案都要sudo但是学生账号没有服务器的sudo权限。

2、解决g++无法解析c++14的问题

该说不说，学校这个服务器应该是被其他学生玩坏了，查了一下/user/bin目录下的gcc版本才4.8.5，害我大半夜在这疯狂找cmake指定gcc的方法。

发现问题在/usr/bin下的gcc版本不够，那么干脆把B19031717文件夹下东西都删了，重新创建了package和下面的bin等四个目录。然后重新cmake编译一下。

执行find / -name "gcc"命令发现gcc文件在/usr/local/bin下也有一个，并且版本是10.0.1，重新cmake编译时记得把DCMAKE_CXX_COMPILER和DCMAKE_C_COMPILER两个路径改成gcc版本高于5.2的路径（mysql-8.0.25需要gcc5.2以上）。

编译未报错，然后执行make -j 64

执行make install

执行cd package然后执行cp -r bin lib share etc ~/与 cd

3、安装、初始化与启动

安装没出现报错，但是初始化时产生了奇怪的问题。貌似是因为升级gcc时，生成的动态库没有替换老版本gcc的动态库。

执行一下strings /usr/lib64/libstdc++.so.6 | grep GLIBC命令，发现如GLIBCXX的版本并不匹配。

执行find / -name "libstdc++.so*"查找编译gcc时生成的最新动态库。发现了/usr/local/lib64/libstdc++.so.6.0.28 //最新动态库

而在上图中使用的库是/usr/lib64/libstdc++.so.6。

我们需要修改系统默认动态库的指向，即：重建默认库的软连接。但是学生账号没有权限，寄。
